# Agent Memory v0.3 Direction

Status: Draft decision document
Branch: `docs/v0.3-architecture-decisions`

## Summary

v0.3 is the integration milestone.

v0.2 made Agent Memory measurable: sessions, memory packs, preflight checks, evidence events, candidates, review, receipts, and deterministic protocol benchmarks. v0.3 should now answer the product question that matters most:

```text
Can real coding agents use Agent Memory naturally enough that it improves behavior without becoming annoying?
```

The direction for v0.3 is therefore **integration-first**, using a generic adapter layer, AGENTS.md-compatible instructions, CLI commands, and a protocol compliance checker.

## Product thesis

Agent Memory is not a notes app, a generic RAG database, or a dashboard.

Agent Memory is a local-first behavior layer for coding agents. It should help agents recall project memory, respect workflow rules, preflight risky actions, capture evidence, propose useful learning, and leave an audit trail.

After v0.3, the product claim should be:

```text
Agent Memory is a local-first behavior layer for coding agents.
```

The technical explanation is:

```text
It gives coding agents an auditable memory protocol: recall, preflight, evidence, reviewed learning, and compliance receipts.
```

## Primary v0.3 decision

v0.3 is **integration-first**.

The milestone is not primarily about retrieval quality, stale/conflict detection, distribution, MCP, or UI. Those matter later, but v0.3 should prove that Agent Memory fits into real agent workflows.

## Target agents

v0.3 should target all coding agents through a **generic adapter layer**.

The first integration surface is still simple:

- root `AGENTS.md`
- Agent Memory CLI
- protocol compliance checker

The adapter layer should avoid locking the design to one agent. Codex, Claude Code, Cursor, and other AGENTS.md-style agents should eventually share the same protocol concepts:

- how the agent is instructed
- how the agent starts a session
- how it loads memory
- how it preflights risky commands
- how it records evidence
- how it proposes candidates
- how it finishes and reports compliance

v0.3 should not implement a full multi-agent adapter system yet. It should design the abstraction and implement the first practical AGENTS.md-backed path.

## Minimum integration surface

v0.3 should provide:

1. AGENTS.md router instructions
2. CLI protocol commands
3. protocol compliance checker

The core new command should be:

```bash
agentmem protocol check --session <session-id> --json
```

It should answer whether the session followed the required protocol checkpoints.

## Required protocol checkpoints

For a complete memory-aware agent session, the protocol should include:

1. start session
2. load pack
3. preflight risky commands
4. record meaningful events when they happen
5. propose memory candidates when reusable learning happens
6. finish session
7. produce receipt / compliance report

The ideal complete flow is:

```text
session_started
pack_loaded
preflight_checked, warning_triggered, or block_triggered when relevant
event_recorded when evidence appears
candidate_proposed when reusable learning appears
session_finished
```

## Every task should use the protocol, without spam

The desired user experience is:

```text
Every task should become memory-aware, but the protocol must not obstruct the agent.
```

That means v0.3 should not define memory use as an occasional, explicit-only workflow. Agents should treat Agent Memory as the default operating mode.

However, this does not mean every tiny edit should produce noisy events, candidates, and questions. The protocol should be adaptive:

- every task should start a session and load a pack
- preflight should run only before risky commands
- events should be recorded only when there is meaningful evidence
- candidates should be proposed only for reusable learning
- uncertainty should be surfaced through manage-memory behavior rather than constant clarification

The intended behavior is:

```text
always memory-aware, rarely noisy
```

## /manage-mem behavior

v0.3 should formalize a manage-memory behavior pattern.

When the agent is not confident whether something should become memory, it should not spam the user during implementation. Instead, it should collect uncertainty and surface a compact manage-memory review point.

The product-level behavior:

- do not ask after every small observation
- do not silently create trusted memory for risky or durable claims
- group memory uncertainty into a concise review moment
- use `manage --plan` or a future `/manage-mem` style instruction as the review surface

This is not necessarily a slash-command implementation in v0.3. It is a behavior requirement for installed router instructions and future adapters.

## Enforcement level

v0.3 should target soft enforcement.

The chosen enforcement model is:

- the agent is instructed to follow the protocol
- the system can check whether it did
- the final output can include or reference a compliance receipt
- missing checkpoints are visible

v0.3 should not implement hard command blocking, command proxying, or policy-engine enforcement.

If the agent skips memory, the system should report it through:

```bash
agentmem protocol check --session <session-id> --json
```

and final receipt/compliance output.

## Compliance report

A v0.3 compliance report should include:

- session started
- pack loaded
- memory IDs injected
- preflight commands checked
- warnings triggered
- blocks triggered
- events recorded
- candidates proposed
- session finished
- missing checkpoints
- overall compliance status

Example shape:

```json
{
  "sessionId": "ses_x",
  "compliant": true,
  "required": {
    "sessionStarted": true,
    "packLoaded": true,
    "sessionFinished": true
  },
  "activity": {
    "memoriesInjected": ["mem_x"],
    "preflightChecks": 1,
    "warningsTriggered": 1,
    "blocksTriggered": 0,
    "eventsRecorded": 1,
    "candidatesProposed": 1
  },
  "missingCheckpoints": []
}
```

## Convenience flow

v0.3 should add a convenience command after compliance checking:

```bash
agentmem protocol start "<task>" --json
```

This command should start a session and generate the initial pack in one step.

It should not replace the existing session and pack commands. It should make agent integration less error-prone.

## Instruction installation

v0.3 should keep installer scope small.

The supported instruction file for v0.3 is:

```text
AGENTS.md
```

The installed router should use a managed block inside existing human-authored content. Agent Memory should not own the entire file.

The preferred model is:

```text
human-authored AGENTS.md
+ small Agent Memory managed block
```

The router should be adaptive:

- simple tasks: start session, load pack, finish receipt
- risky commands: preflight before command
- meaningful evidence: record event
- reusable lesson: propose candidate
- uncertainty: defer to manage-memory review

## Dogfooding

The `agent-memory` repo itself should become the first dogfood target, but only for non-trivial PRs.

Dogfood requirement:

```text
Future non-trivial PRs should have a memory session and a compliance receipt.
```

This should slow development slightly if needed. The point of v0.3 is to prove the product changes real agent behavior.

## Low-risk active memory

v0.3 may allow agents to create active memory for low-risk observations, but this must be tightly constrained.

Default rule:

- agents can propose durable candidates
- users review important memory

Possible v0.3 exception:

- agent may create active memory for low-risk, non-secret, non-policy, non-architectural observations
- the memory must be evidence-backed
- the source must clearly remain `agent_reported`
- the receipt must show that the agent-created active memory happened
- command policies, architecture decisions, workflow rules, failures, fixes, and user preferences should still require review

This is an explicit open design area for v0.3. It should not be implemented until the protocol compliance checker exists.

## Candidate review

v0.3 should not focus on candidate review UX.

Current candidate commands and `manage --plan` are sufficient for v0.3. Candidate grouping, merge/supersede, and interactive review are deferred.

## Retrieval work allowed in v0.3

Retrieval is not the v0.3 focus.

Allowed:

- small pack formatting improvements
- better source/authority display
- returning both markdown and structured JSON if needed by agents

Deferred:

- FTS5/BM25
- vector search
- stale/conflict-aware retrieval
- full retrieval redesign

## Pack format direction

Memory packs should eventually return both:

- agent-readable markdown
- structured JSON

v0.3 may improve `pack --json` if the compliance checker or adapter layer needs richer structure.

Do not redesign retrieval until dogfood and benchmarks identify concrete retrieval failures.

## Explicit v0.3 non-goals

Do not implement these in v0.3 unless this decision document is changed:

- MCP server
- vector search
- cloud sync
- dashboard/UI
- broad multi-agent adapter implementation
- command proxy / hard command blocking
- stale/conflict engine
- full retrieval redesign

## Success metrics

v0.3 succeeds when all of the following are true:

1. Codex or another real coding agent follows the protocol on at least three real non-trivial PRs.
2. `agentmem protocol check --session <id>` passes on those PRs.
3. The agent avoids at least one known mistake because memory was injected.
4. The agent proposes at least one useful candidate from real work.
5. The final receipt makes the memory behavior auditable.

## First implementation slice

The first implementation PR after these docs should be:

```text
agentmem protocol check --session <session-id>
```

This comes before MCP, FTS5, stale detection, or pack redesign.

## Open questions for later

- What exact memory types are safe for low-risk active agent creation?
- Should `protocol check` be strict for tiny tasks or adaptive by task complexity?
- Should `protocol start` be part of v0.3 or the second v0.3 slice?
- Should adapter profiles be data files or hardcoded behavior?
- Should future AGENTS.md installation support CLAUDE.md and Cursor rules through the same adapter abstraction?
