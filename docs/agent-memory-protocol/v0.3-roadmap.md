# Agent Memory v0.3 Roadmap

Status: Draft roadmap

## Milestone

v0.3 is the real-agent integration milestone.

v0.2 proved the local protocol primitives. v0.3 should prove that a coding agent can use those primitives reliably in a real repo workflow without turning memory into busywork.

## Goal

Build the integration layer that makes Agent Memory the default behavior layer for coding agents.

The goal is not to make memory more intelligent first. The goal is to make memory reliably used.

## Product outcome

After v0.3, a repo should be able to install Agent Memory instructions and have an agent naturally follow this flow:

```text
start memory-aware session
load relevant pack
preflight risky commands
record meaningful evidence
propose useful candidates
finish with a compliance receipt
```

The user should be able to verify the behavior with:

```bash
agentmem protocol check --session <session-id> --json
```

## Non-goals

Do not implement in v0.3:

- MCP server
- vector search
- cloud sync
- dashboard/UI
- broad multi-agent adapter implementation
- command proxy or hard blocking
- stale/conflict engine
- full retrieval redesign
- interactive memory-management UI

## Phase 1: Protocol compliance checker

Add:

```bash
agentmem protocol check --session <session-id> --json
agentmem protocol check --session <session-id>
```

Responsibilities:

- read session receipts
- report required checkpoints
- report optional activity
- show missing checkpoints
- return a deterministic compliance result
- be useful in final agent responses and dogfood PR reviews

Suggested JSON shape:

```json
{
  "sessionId": "ses_x",
  "compliant": true,
  "required": {
    "sessionStarted": true,
    "packLoaded": true,
    "sessionFinished": true
  },
  "activity": {
    "memoriesInjected": ["mem_x"],
    "preflightChecks": 1,
    "warningsTriggered": 1,
    "blocksTriggered": 0,
    "eventsRecorded": 1,
    "candidatesProposed": 1,
    "candidatesReviewed": 0
  },
  "missingCheckpoints": []
}
```

Compliance should be soft. It should report missing checkpoints, not block work.

## Phase 2: Protocol start convenience command

Add:

```bash
agentmem protocol start "<task>" --json
```

Responsibilities:

- start a session
- generate an initial memory pack
- return session ID, pack markdown, and matched memory IDs
- reduce the number of steps agents must remember

This command should not replace `session start` or `pack`; it is a convenience wrapper for agent integrations.

## Phase 3: AGENTS.md router hardening

Improve installed router instructions for v0.3 behavior.

Responsibilities:

- keep the managed block small
- support adaptive behavior
- require protocol use for every task while avoiding noisy events/candidates
- explain when to preflight commands
- explain when to record evidence
- explain when to propose candidates
- explain when to defer uncertainty to manage-memory review

The router should make the default behavior:

```text
always memory-aware, rarely noisy
```

## Phase 4: Generic adapter design

Design, but do not overbuild, a generic adapter model.

Potential adapter concepts:

- adapter name
- instruction target file
- managed block format
- startup command
- pack command
- compliance command
- final receipt instruction

v0.3 should implement only the AGENTS.md-backed adapter path unless another agent target becomes unavoidable.

## Phase 5: Dogfood protocol on real PRs

Use the `agent-memory` repo itself as the first dogfood target for non-trivial PRs.

Dogfood rules:

- use Agent Memory for non-trivial implementation PRs
- include the session receipt or protocol check summary in the final handoff
- record evidence for meaningful test/build results or user corrections
- propose candidates only for reusable learning

Success is not that every task produces candidates. Success is that the protocol is followed and does not feel intrusive.

## Phase 6: Small pack output improvements

Only after compliance checking exists, improve pack output if integration work shows a need.

Allowed improvements:

- clearer sections
- source/confidence/severity display
- structured JSON returned alongside markdown
- better distinction between required rules, relevant memories, and warnings

Do not add FTS5 or redesign retrieval in v0.3 unless benchmarks and dogfood demonstrate retrieval is the blocking issue.

## Phase 7: Low-risk active memory design spike

Investigate whether agents should create active memory for low-risk observations.

Constraints:

- no secrets
- no user preferences
- no command policies
- no architecture decisions
- no failed attempts or known fixes without review
- must be evidence-backed
- must be auditable through receipts
- source must remain `agent_reported`

This should start as a design spike or experimental flag, not default behavior.

## Acceptance criteria

v0.3 is complete when:

- `agentmem protocol check --session <id>` exists
- compliance output identifies missing checkpoints
- installed AGENTS.md instructions describe the v0.3 protocol clearly
- a convenience protocol start flow exists or is explicitly deferred with reason
- at least three real non-trivial dogfood PRs have protocol compliance reports
- at least one dogfood task shows memory changing agent behavior
- at least one useful candidate is proposed from real work
- v0.3 does not introduce MCP, vectors, cloud sync, dashboard, or hard blocking

## First implementation PR

The first v0.3 implementation PR should be:

```text
Implement protocol compliance checker
```

Suggested files:

```text
src/core/protocol-check.ts
src/formatters/protocol-check.ts
src/cli/main.ts
tests/protocol-check.test.ts
tests/protocol-check-cli.test.ts
```

Suggested command:

```bash
agentmem protocol check --session <session-id> --json
```

## Definition of success

v0.3 succeeds if Agent Memory becomes a natural part of the coding-agent loop.

The final question is:

```text
Did a real agent follow memory protocol, behave better, and leave proof?
```
