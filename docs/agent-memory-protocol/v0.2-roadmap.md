# Agent Memory v0.2 Roadmap

This roadmap defines the next product milestone after the clean internal CLI baseline.

The milestone is not embeddings, MCP, cloud sync, or a UI.

The milestone is the Agent Memory Protocol: a lightweight, portable behavior layer that makes memory visible, auditable, and useful inside coding-agent workflows.

## Goal

Build a local-first memory protocol that lets an agent:

1. start a memory-aware session
2. load relevant memory before planning
3. preflight risky commands
4. record meaningful events
5. propose memory candidates
6. review/manage candidates and conflicts
7. produce a final memory receipt

## Non-goals

Do not implement in v0.2:

- embeddings/vector sidecar
- graph memory
- MCP server
- cloud sync
- hosted dashboard
- cross-repo global memory
- automatic long-term memory writes without review
- hard command blocking unless trivial through existing CLI behavior

## Proposed commands

### Install protocol docs/instructions

```bash
agentmem install
agentmem install-instructions
agentmem uninstall-instructions
agentmem doctor
```

Responsibilities:

- create or update managed instruction blocks
- install router docs
- show whether the repo is correctly memory-aware
- avoid overwriting user content

### Session lifecycle

```bash
agentmem session start "<task>" --json
agentmem session receipt --session <session-id> --json
agentmem session finish --session <session-id> --summary "..." --json
```

Responsibilities:

- create session IDs
- attach memory events to a session
- produce auditable receipts

### Recall

```bash
agentmem pack "<task>" --session <session-id> --files <paths> --json
```

Enhancements over current pack:

- log session-aware retrieval receipts
- include memory IDs, authority, confidence, and source
- distinguish pinned/core memory from retrieved/task memory

### Preflight

```bash
agentmem preflight --command "<command>" --session <session-id> --json
```

Future:

```bash
agentmem preflight --file "<path>" --operation edit --session <session-id> --json
```

Responsibilities:

- warn/block based on command policies
- log preflight receipts
- later: file/path warnings for fragile files

### Event recording

```bash
agentmem event record --session <session-id> --type test_result --json
agentmem event record --session <session-id> --type command_result --json
agentmem event record --session <session-id> --type user_correction --json
```

Responsibilities:

- record evidence
- do not automatically create durable memory
- provide source material for candidates

### Candidate proposal

```bash
agentmem candidate propose --session <session-id> --type failed_attempt --content "..." --evidence "..." --json
agentmem candidate propose --session <session-id> --type known_fix --content "..." --evidence "..." --json
agentmem candidate propose --session <session-id> --type agent_mistake --content "..." --evidence "..." --json
agentmem candidate list --json
agentmem candidate approve <candidate-id> --json
agentmem candidate reject <candidate-id> --reason "..." --json
```

Responsibilities:

- separate agent-proposed candidates from trusted memory
- require source evidence
- support approval/rejection later

### Manage mode

```bash
agentmem manage --plan
```

Responsibilities:

- show candidates
- show conflicts
- show possible stale memories
- ask concise review questions
- promote/reject/merge/supersede memory

## Suggested data model additions

### `sessions`

Tracks agent runs.

Fields:

- `session_id`
- `project_id`
- `task`
- `status`
- `started_at`
- `finished_at`
- `summary`

### `memory_candidates`

Stores untrusted proposed memory.

Fields:

- `candidate_id`
- `project_id`
- `session_id`
- `type`
- `content`
- `scope`
- `source`
- `confidence`
- `severity`
- `evidence`
- `candidate_status`
- `proposed_by`
- `created_at`
- `reviewed_at`
- `review_reason`
- `target_memory_id` for merge/supersede flows

Candidate statuses:

- `proposed`
- `approved`
- `rejected`
- `merged`
- `superseded`
- `expired`

### `protocol_receipts`

Stores auditable proof of protocol usage.

Fields:

- `receipt_id`
- `project_id`
- `session_id`
- `receipt_type`
- `payload_json`
- `created_at`

Receipt types:

- `session_started`
- `pack_loaded`
- `preflight_checked`
- `warning_triggered`
- `block_triggered`
- `candidate_proposed`
- `candidate_reviewed`
- `stale_detected`
- `session_finished`

### `repo_state_snapshots`

Stores lightweight repo state used for stale detection.

Fields:

- `snapshot_id`
- `project_id`
- `session_id`
- `git_head`
- `package_manager`
- `lockfiles_json`
- `test_commands_json`
- `dependency_manifest_hash`
- `created_at`

## Implementation phases

### Phase 1: Protocol docs and installer

- Add managed instruction block generator.
- Add `agentmem install-instructions`.
- Add `agentmem doctor`.
- Add protocol docs under `.agent-memory/protocol.md` or repo docs.
- Verify no user content is overwritten.

### Phase 2: Sessions and receipts

- Add `sessions` table.
- Add `protocol_receipts` table.
- Add `session start`, `session finish`, and `session receipt` commands.
- Attach `pack` and `preflight` calls to sessions.

### Phase 3: Candidate memory lifecycle

- Add `memory_candidates` table.
- Add candidate propose/list/approve/reject.
- Add candidate hygiene checks:
  - no secrets
  - no one-off details
  - no obvious repo facts
  - require evidence for agent-proposed candidates

### Phase 4: Manage mode

- Add `agentmem manage --plan`.
- Show candidates, conflicts, stale signals, and duplicates.
- Keep output concise and plan-mode friendly.

### Phase 5: FTS5 retrieval upgrade

- Add FTS index for memory content, summary, tags, paths, and evidence.
- Keep deterministic filters first.
- Use FTS/BM25 before vector search.

### Phase 6: Stale/conflict detection basics

- Detect missing file paths.
- Detect package manager/lockfile changes.
- Detect newer conflicting memories by type/scope/path.
- Mark stale candidates, not destructive changes by default.

## Acceptance criteria for v0.2

The milestone is complete when:

- repo instruction router can be installed safely
- agent can start a session and get a session ID
- pack/preflight calls can attach to a session
- final receipt can show actual memory tool usage
- agent can propose memory candidates without directly trusting them
- manage mode can review candidates at a high level
- benchmark fixtures can test protocol compliance, not just retrieval

## Definition of success

Agent Memory v0.2 succeeds if the system can answer:

- Did the agent load memory before planning?
- Which memory IDs were included?
- Did the agent preflight risky commands?
- Did warnings or blocks trigger?
- Did the agent propose useful memory candidates after failures/fixes/corrections?
- Did memory usage produce an auditable receipt?

The product should become measurable before it becomes more intelligent.
