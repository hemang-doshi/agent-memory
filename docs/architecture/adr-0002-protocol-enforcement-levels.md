# ADR 0002: v0.3 Protocol Enforcement Levels

Status: Accepted for v0.3 planning
Date: 2026-06-27

## Context

Agent Memory needs to influence agent behavior without becoming obstructive.

The user wants the protocol to apply to every task, including tiny edits, but without spam. This creates a tension:

```text
always memory-aware
rarely noisy
```

v0.3 must make protocol use visible and checkable while avoiding hard blocking, excessive prompts, or forced memory creation.

## Decision

v0.3 will use soft enforcement.

The target enforcement model is:

1. instruction-level guidance through AGENTS.md
2. actual receipts from CLI/core behavior
3. protocol compliance checker
4. final receipt/compliance summary

v0.3 will not implement hard command blocking, command proxying, or an external policy engine.

## Enforcement levels

### Level 1: Instruction

The agent is told how and when to use memory.

This is implemented through the managed AGENTS.md block.

### Level 2: Receipt

The system records what actually happened:

- session started
- pack loaded
- preflight checked
- warning/block triggered
- event recorded
- candidate proposed
- candidate reviewed
- session finished

Receipts are generated by core behavior, not agent self-report.

### Level 3: Compliance check

`agentmem protocol check --session <id>` reads receipts and reports whether the expected protocol happened.

This is the core v0.3 enforcement mechanism.

### Level 4: Final response discipline

Agents should include a compact receipt or compliance summary in final handoffs for non-trivial tasks.

### Deferred: Hard enforcement

Hard blocking, shell wrappers, command proxies, or policy engines are deferred.

## Required vs adaptive checkpoints

Every task should start memory-aware, but not every task needs every protocol action.

### Required for every task

- session exists
- pack loaded
- session is finished or explicitly still active

### Required when applicable

- preflight before risky commands
- event recording when meaningful evidence appears
- candidate proposal when reusable learning appears

### Never required by default

- proposing candidates for one-off work
- recording events for trivial observations
- asking the user after every possible memory

## Compliance report behavior

A protocol check should report:

- compliant: boolean
- missing required checkpoints
- memory IDs injected
- preflight count
- warning/block count
- event count
- candidate count
- session status

A missing optional checkpoint should not fail compliance unless the fixture or command explicitly requires it.

Example:

```text
A tiny typo fix with session start, pack load, and finish can be compliant with zero candidates.
```

## Handling skipped memory

If the agent skips memory, v0.3 should not halt work retroactively. Instead:

- protocol check reports the missing checkpoint
- final receipt shows non-compliance
- dogfood review can catch repeated misses

This creates accountability without requiring a command proxy.

## Manage-memory behavior

When the agent is uncertain whether something should become memory, it should not interrupt constantly.

Instead:

- collect uncertainty during the task
- propose candidates only for clear reusable learning
- surface ambiguous items through manage-memory review

This supports the desired behavior:

```text
useful enforced addition, not intrusive obstruction
```

## Low-risk active memory

v0.3 may later permit agent-created active memory for low-risk observations, but this is outside the first enforcement slice.

If implemented, it must obey strict constraints:

- no secrets
- evidence-backed
- low-risk type only
- `agent_reported` source
- auditable receipt
- no command policies, architecture decisions, user preferences, failed attempts, or known fixes without review

The compliance checker should exist before this exception is implemented.

## Consequences

Positive:

- agents can be held accountable without hard blocking
- protocol use becomes measurable
- tiny tasks can stay lightweight
- the product remains local-first and portable

Negative:

- agents may still skip steps unless final checks are used
- soft enforcement depends on dogfood discipline
- hard safety guarantees are deferred

## Alternatives considered

### Pure instruction only

Rejected. Instructions alone are too easy for agents to skip or fake.

### Hard command blocking

Rejected for v0.3. It is too invasive before integration behavior is proven.

### Every checkpoint required for every task

Rejected. It would create spam and make memory feel obstructive.

- implement `agentmem protocol check`
- update AGENTS.md router text to explain adaptive requirements
- add dogfood examples showing compliant tiny and non-trivial tasks
